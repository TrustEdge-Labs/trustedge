<interfaces>
<!-- The verify page calls this endpoint. From Phase 76: -->

POST /v1/verify-attestation
- Request body: raw attestation JSON (the .te-attestation.json file contents)
- Response 200 (verified): { "status": "verified", "receipt": "<JWS>", "details": {...} }
- Response 200 (failed): { "status": "failed", "details": {...} }
- Response 400: { "error": "invalid_attestation"|"invalid_format"|"missing_signature" }
- Response 429: rate limited
- Response 413: body too large (>2MB)

The attestation JSON contains:
- format: "te-point-attestation-v1"
- public_key: "ed25519:base64..." (embedded, used for self-contained verification)
- subject: { label: "binary", blake3_hash: "...", size_bytes: N }
- evidence: [{ label: "sbom", blake3_hash: "...", size_bytes: N, content: "...base64..." }]
- nonce: "base64..."
- timestamp: "ISO8601"
- signature: "base64..."
</interfaces>

Create `web/verify/index.html` — a single self-contained HTML file (~100-150 lines) with inline CSS and JS. No external dependencies, no build step.

**Layout (per D-01):**
- Clean white background, professional appearance (per specifics: "feel like a professional tool page")
- Page title: "TrustEdge Attestation Verifier"
- File input accepting `.json` files, labeled "Select .te-attestation.json"
- "Verify" button (disabled until file selected)
- Result area (hidden until verification attempted)

**JS logic (per D-03):**
- On "Verify" click: read the file as text, POST to `/v1/verify-attestation` (relative URL, same origin) with the raw JSON text as body
- Set Content-Type: text/plain (the endpoint uses String extractor, not Json)
- AbortController with 30-second timeout (per D-04)
- On success (status=verified): display receipt details:
  - Verification status (green checkmark with "Verified")
  - Subject (binary) BLAKE3 hash (monospace font)
  - Evidence (SBOM) details — parse the attestation JSON client-side to show label, hash, size
  - Signing public key (monospace)
  - Timestamp
  - Format version
- On success (status=failed): display "Verification Failed" in red with details
- On network error: show "Server unavailable, please try again later" (per D-04)
- On timeout: show "Request timed out" (per D-04)
- On 400 error: parse error body and show appropriate message
- On 429: show "Rate limited — please wait and try again"
- On 413: show "File too large (max 2 MB)"

**Styling:**
- Monospace font for hashes and keys (font-family: monospace)
- Green (#22c55e) for verified, red (#ef4444) for failed
- Card-style result container with subtle border
- Responsive: works on mobile (max-width container, padding)
- TrustEdge UTF-8 symbols: ✔ for pass, ✖ for fail

**Third-party flow (per D-05 override from planning context):**
- The embedded public key in the attestation is used for verification — no separate key upload needed
- This satisfies PLAT-03: the verifier doesn't need their own key, the attestation is self-contained

**Do NOT:**
- Add a separate public key upload field (deferred per D-05)
- Add a binary file upload for hash comparison (deferred)
- Use any external CDN, framework, or build tool
- Add animations or heavy branding
