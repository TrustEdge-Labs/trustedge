**SEC-11: Additional truncation boundaries (beyond SEC-08)**

1. `sec_11_truncated_single_byte_ciphertext` — Take a valid key file, find the ciphertext start (after second newline), keep only 1 byte of ciphertext. Call `import_secret_encrypted` with correct passphrase. Assert `DecryptionFailed` containing "Wrong passphrase". This tests that AES-GCM rejects ciphertext shorter than the 16-byte auth tag.

2. `sec_11_truncated_at_gcm_tag_boundary` — Take a valid key file, truncate to keep exactly 16 bytes of ciphertext (the minimum for an AES-GCM tag with no plaintext). Call `import_secret_encrypted` with correct passphrase. Assert `DecryptionFailed` containing "Wrong passphrase". The tag will not validate because the plaintext portion is missing.

**SEC-12: Corrupted JSON fields (beyond SEC-09)**

3. `sec_12_json_version_wrong_type` — Build a key file with `{"version":"not-a-number","salt":"<valid>","nonce":"<valid>","iterations":600000}`. The version field has wrong type (string instead of integer). Since `import_secret_encrypted` does not currently validate version, this should still attempt decryption — assert `DecryptionFailed` containing "Wrong passphrase" (the dummy ciphertext will fail AEAD). This confirms version field is tolerated when present with wrong type (backward compat).

4. `sec_12_json_iterations_wrong_type` — Build a key file with `{"salt":"<valid>","nonce":"<valid>","iterations":"not-a-number"}`. Since `iterations` is read via `.as_u64()`, a string value returns `None`. Assert `InvalidKeyFormat` containing "Missing iterations".

5. `sec_12_json_nonce_bad_base64` — Build a key file with valid salt, `"nonce":"$$invalid$$"`, valid iterations. Assert `InvalidKeyFormat` containing "Invalid nonce base64".

6. `sec_12_json_salt_wrong_length` — Build a key file with `"salt":"<base64 of 16 bytes>"` (16 bytes instead of 32). The PBKDF2 will use the short salt, derive a different key, and AES-GCM will reject. Assert `DecryptionFailed` containing "Wrong passphrase". This confirms short salts don't cause panics.

7. `sec_12_json_extra_unknown_fields` — Build a key file with valid salt, nonce, iterations, PLUS `"future_field":"ignored"`. Call `import_secret_encrypted` with the correct passphrase. This should still fail with `DecryptionFailed` (because the ciphertext is dummy), but the key point is it does NOT fail with `InvalidKeyFormat` — unknown fields are tolerated. Assert `DecryptionFailed` containing "Wrong passphrase".

For tests 3-7, use `build_corrupted_key_file(meta_json, b"dummy-ciphertext")` with crafted JSON strings. Use `BASE64.encode([0u8; 32])` for valid salt and `BASE64.encode([0u8; 12])` for valid nonce, matching existing SEC-09 patterns.
