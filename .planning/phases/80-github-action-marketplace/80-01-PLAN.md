Rewrite the "Download trst binary" step in `actions/attest-sbom-action/action.yml` to
add SHA256 checksum verification (per D-04). The checksum file asset name is `trst.sha256`
and lives alongside the binary on each GitHub Release.

Replace the existing download step with this logic:

```bash
VERSION="${{ inputs.trst-version }}"
if [ "$VERSION" = "latest" ]; then
  BASE_URL="https://github.com/TrustEdge-Labs/trustedge/releases/latest/download"
else
  BASE_URL="https://github.com/TrustEdge-Labs/trustedge/releases/download/${VERSION}"
fi

# Download binary
curl -fsSL "${BASE_URL}/trst" -o "${{ runner.temp }}/trst"
chmod +x "${{ runner.temp }}/trst"

# Download and verify SHA256 checksum (per D-04: graceful degradation if not present)
if curl -fsSL "${BASE_URL}/trst.sha256" -o "${{ runner.temp }}/trst.sha256" 2>/dev/null; then
  # The checksum file contains "HASH  filename" — extract just the hash
  EXPECTED_HASH=$(awk '{print $1}' "${{ runner.temp }}/trst.sha256")
  ACTUAL_HASH=$(sha256sum "${{ runner.temp }}/trst" | awk '{print $1}')
  if [ "$EXPECTED_HASH" != "$ACTUAL_HASH" ]; then
    echo "SHA256 mismatch: expected $EXPECTED_HASH, got $ACTUAL_HASH"
    exit 1
  fi
  echo "SHA256 verified: $ACTUAL_HASH"
else
  echo "Warning: No SHA256 checksum file found for this release. Skipping verification."
fi
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Releases to action runner | Binary downloaded over HTTPS; SHA256 checksum downloaded from same release |
| action repo to user's CI pipeline | Users `uses: TrustEdge-Labs/attest-sbom-action@v1` — tag pinning prevents substitution |
| user's GitHub Secret to runner env | TRUSTEDGE_KEY secret decoded at runtime via base64; key material visible in runner memory |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-80-01 | Tampering | Binary download in action.yml | mitigate | SHA256 checksum verified before execution (per D-04); mismatched hash calls `exit 1` |
| T-80-02 | Tampering | SHA256 checksum file itself | accept | Checksum file fetched from same GitHub Release as binary; Release assets are immutable once published; key signing of checksums is out of scope |
| T-80-03 | Elevation of Privilege | `gh repo create` requires org write access | mitigate | Only org admins can create repos under TrustEdge-Labs; executor runs as gh-authenticated user; documented in user_setup |
| T-80-04 | Information Disclosure | Persistent key stored as GitHub Secret | accept | GitHub Secrets are write-only in the UI, never echoed in logs; standard GitHub Actions security model |
| T-80-05 | Spoofing | Tag v1 could be force-pushed to a different commit | accept | Pinning to commit SHA (e.g., `@abc1234`) is more robust; acceptable for v1 scope — recommend SHA pinning in README security note |
| T-80-06 | Denial of Service | Missing trst.sha256 in older releases causes action to skip verification | accept | Graceful degradation per D-04 — warning printed, execution continues. Only affects users who pin to pre-checksum releases |
</threat_model>
