Add request body size limiting (2 MB global) and per-IP rate limiting on /v1/verify to the platform HTTP layer.

Purpose: Protect against body-flood DoS (OOM) and CPU-exhaustion abuse of the BLAKE3+Ed25519 verify endpoint.
Output: RequestBodyLimitLayer on all routes, governor-based per-IP rate limiter on /v1/verify only, integration tests proving 413 and 429 responses.
</objective>

**Test 1: test_body_limit_413** — Send a POST to `/v1/verify` with a body larger than 2 MB (e.g., 3 MB of zeros as a JSON string). Assert response status is 413 Payload Too Large.

```rust
#[tokio::test]
async fn test_body_limit_413() {
    let app = create_test_app().await;
    // Build a body > 2 MB
    let large_body = "x".repeat(3 * 1024 * 1024);
    let body = format!(r#"{{"manifest":{{"data":"{}"}}, "segments":[], "device_pub":"ed25519:test"}}"#, large_body);
    let req = Request::builder()
        .method("POST")
        .uri("/v1/verify")
        .header(CONTENT_TYPE, "application/json")
        .body(Body::from(body))
        .unwrap();
    let resp = app.oneshot(req).await.unwrap();
    assert_eq!(resp.status(), 413);
}
```

**Test 2: test_body_under_limit_not_413** — Send a POST to `/v1/verify` with a small valid body (< 2 MB). Assert response is NOT 413 (it may be 400 due to invalid content, but not 413).

**Test 3: test_rate_limit_429** — Send requests to `/v1/verify` in rapid succession exceeding the default rate limit. Since the test app uses `create_router()` which reads `RATE_LIMIT_RPS` from env, set `RATE_LIMIT_RPS=2` before creating the app (or use a low value). Send 10+ requests in a tight loop. Assert that at least one response returns 429.

IMPORTANT: The test app created via `create_test_app()` -> `create_router()` does NOT have `ConnectInfo` populated because `tower::ServiceExt::oneshot` doesn't go through `into_make_service_with_connect_info`. Two options:
- Option A: The rate limit middleware should handle the case where `ConnectInfo` is not available by falling back to a default IP (e.g., `127.0.0.1`). This is the preferred approach — add a fallback in the middleware.
- Option B: Use `axum_test` crate which simulates real connections.

Prefer Option A: modify `rate_limit.rs` to use `Option<ConnectInfo<SocketAddr>>` and fall back to `127.0.0.1` if not available. This makes the middleware testable AND production-safe.

If Option A is used, update rate_limit_middleware signature to accept `connect_info: Option<ConnectInfo<SocketAddr>>` and extract IP as:

let ip = connect_info.map(|ci| ci.0.ip()).unwrap_or(IpAddr::V4(std::net::Ipv4Addr::LOCALHOST));
