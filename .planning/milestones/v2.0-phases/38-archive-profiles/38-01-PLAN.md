Refactor `crates/trst-protocols/src/archive/manifest.rs`:

1. Rename `CamVideoManifest` to `TrstManifest`. Add `pub type CamVideoManifest = TrstManifest;` alias.

2. Replace the `capture: CaptureInfo` field with `metadata: ProfileMetadata`. The `ProfileMetadata` enum should be:
```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(untagged)]
pub enum ProfileMetadata {
    Generic(GenericMetadata),
    CamVideo(CamVideoMetadata),
}
```

3. Define `GenericMetadata`:
```rust
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub struct GenericMetadata {
    pub started_at: String,
    pub ended_at: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub data_type: Option<String>,      // e.g., "video", "sensor", "audio", "log", "binary"
    #[serde(skip_serializing_if = "Option::is_none")]
    pub source: Option<String>,         // e.g., "drone-cam-01"
    #[serde(skip_serializing_if = "Option::is_none")]
    pub description: Option<String>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub mime_type: Option<String>,
    #[serde(default, skip_serializing_if = "BTreeMap::is_empty")]
    pub labels: BTreeMap<String, String>,  // BTreeMap for sorted keys in canonical output
}
```

4. Define `CamVideoMetadata`:
```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CamVideoMetadata {
    pub started_at: String,
    pub ended_at: String,
    pub timezone: String,
    pub fps: f64,
    pub resolution: String,
    pub codec: String,
}
```

Note: `started_at` and `ended_at` are common to both profiles but live inside each metadata variant (not extracted to base) to keep canonical serialization straightforward. The `CaptureInfo` struct can be kept as a deprecated alias for `CamVideoMetadata` if needed for compilation, but the primary type is `CamVideoMetadata`.

5. Keep `CaptureInfo` as `pub type CaptureInfo = CamVideoMetadata;` for backward compatibility.

6. Update `to_canonical_bytes()` to dispatch on profile type:
   - For "cam.video": serialize metadata fields in the same order as the old `capture` section (started_at, ended_at, timezone, fps, resolution, codec) under the key `"metadata"`.
   - For "generic": serialize metadata fields in order: started_at, ended_at, data_type (if present), source (if present), description (if present), mime_type (if present), labels (if non-empty, with sorted keys) under the key `"metadata"`.
   - Root key order: trst_version, profile, device, metadata, chunk, segments, claims, prev_archive_hash (if present).

IMPORTANT: The serde `#[serde(untagged)]` on `ProfileMetadata` means deserialization tries each variant in order. Put `CamVideo` first since it has required fields (timezone, fps, resolution, codec) that disambiguate it from Generic. If a manifest has those fields, it deserializes as CamVideo; otherwise as Generic.

Actually, a better approach than untagged: since the manifest already has a `profile` field, implement custom deserialization or use a helper method. The simplest approach: use `#[serde(untagged)]` but order variants so CamVideo (more specific) is tried first. OR: deserialize the metadata as `serde_json::Value` first, then dispatch based on the `profile` field in a `TrstManifest::from_json()` helper. However, for simplicity with serde, the untagged approach with CamVideo first should work since CamVideo has required fields that Generic does not.

7. Update `validate()`:
   - Remove the `profile != "cam.video"` rejection (lines 260-263).
   - Accept "generic" and "cam.video" as valid profile values.
   - For cam.video: validate started_at, ended_at are non-empty (from CamVideoMetadata).
   - For generic: validate started_at, ended_at are non-empty (from GenericMetadata).
   - Keep existing device and segment validation.

8. Update `TrstManifest::new()` to create a generic profile by default (profile="generic", metadata=ProfileMetadata::Generic(GenericMetadata::default())).

9. Add `TrstManifest::new_cam_video()` constructor that creates a cam.video profile manifest with CamVideoMetadata defaults.

10. Update existing tests to use the new types. Keep test coverage for canonical byte ordering.

In `crates/trst-protocols/src/capture/profile.rs`: no changes needed beyond what already exists.
