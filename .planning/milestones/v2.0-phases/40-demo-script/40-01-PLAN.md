1. In crates/trst-cli/src/main.rs, add a `Keygen` variant to the `Commands` enum and a `KeygenCmd` struct:

```rust
Keygen(KeygenCmd),
```

```rust
#[derive(Args, Debug)]
struct KeygenCmd {
    #[arg(long = "out-key", value_name = "PATH", help = "Output path for secret key file")]
    out_key: PathBuf,
    #[arg(long = "out-pub", value_name = "PATH", help = "Output path for public key file")]
    out_pub: PathBuf,
}
```

2. Add `Commands::Keygen(args) => handle_keygen(args)` to the match in `run()`. Since handle_keygen is sync, wrap it as the other sync commands do (return Result directly).

3. Implement `handle_keygen`:
   - Check if out_key or out_pub already exist. If either does, bail with "Refusing to overwrite existing file: {path}".
   - Call `DeviceKeypair::generate()?`.
   - Write secret to out_key: `format!("{}\n", device_keypair.export_secret())`.
   - Write public to out_pub: `format!("{}\n", device_keypair.public)`.
   - Print: "Generated device key: {out_key}" and "Generated device pub: {out_pub}".

4. In acceptance.rs, add three tests:
   - `acceptance_keygen_creates_files`: runs keygen, asserts both files exist and start with "ed25519:".
   - `acceptance_keygen_roundtrip`: keygen -> wrap with --device-key/--device-pub -> verify with device pub string. Full round-trip passes.
   - `acceptance_keygen_no_overwrite`: create a file at out_key path, run keygen, assert failure with stderr containing "overwrite".

Note: The existing `load_or_generate_keypair` function already demonstrates the exact file format. Keygen just makes this an explicit user-facing command with explicit output paths (no auto-detection of device.key/device.pub in CWD).
