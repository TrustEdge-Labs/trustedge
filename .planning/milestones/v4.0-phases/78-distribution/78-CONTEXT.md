# Phase 78: Distribution - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Final phase of v4.0. Deliver four things: (1) landing page content for the trustedgelabs-website, (2) self-attestation CI step in the release workflow, (3) third-party attestation documentation, and (4) a GitHub Action for one-line CI integration.

This phase makes the SBOM attestation wedge discoverable and usable by other projects.

</domain>

<decisions>
## Implementation Decisions

### Landing Page
- **D-01:** Content file in the monorepo at `docs/landing-page.md`, NOT a deployed page in the trustedgelabs-website repo. User deploys to their website separately on their own schedule.
- **D-02:** Content includes: product headline, what it does (1-2 sentences), 3-command quick start, link to public verifier URL, link to GitHub Action, link to GitHub repo.
- **D-03:** Positioning per CEO review: infrastructure-independent verification, cross-CI, hardware key support. Differentiation vs GitHub Attestations and Sigstore.

### Self-Attestation CI Step
- **D-04:** Add a step to the existing release workflow (`.github/workflows/ci.yml`, triggered on `v*` tags) that attests the release binary.
- **D-05:** Ephemeral key: CI generates a throwaway Ed25519 keypair per run using `trst keygen --unencrypted`. No persistent key in GitHub Secrets. The public key is embedded in the attestation document.
- **D-06:** Output: `.te-attestation.json` uploaded as a GitHub Release asset alongside the release binary.
- **D-07:** The attestation signs the built `trst` binary + a CycloneDX SBOM generated by `syft` in the same CI step.

### Third-Party Demo
- **D-08:** Documentation only (not an actual attestation of another project). A README section or docs page showing how any project can use `trst attest-sbom` to attest their own releases.
- **D-09:** Include a copy-paste example: keygen → attest-sbom → verify-attestation → upload to release. Covering both manual and CI workflows.

### GitHub Action
- **D-10:** New repo: `TrustEdge-Labs/attest-sbom-action` on the TrustEdge-Labs GitHub org. Required by GitHub for marketplace publishing.
- **D-11:** Composite action (action.yml with `runs: using: composite`). No Docker. Downloads pre-built `trst` binary from the TrustEdge GitHub Releases.
- **D-12:** Inputs: `binary` (path to artifact), `sbom` (path to CycloneDX JSON), `key` (optional, generates ephemeral if not provided). Output: `attestation-path` (path to generated .te-attestation.json).
- **D-13:** The action itself is a separate deliverable. For this phase, create the action.yml, README, and LICENSE in a local directory (`actions/attest-sbom-action/`). User pushes to the new repo and tags v1 manually.

### Claude's Discretion
- Landing page copy tone and structure
- Whether to generate the SBOM in the CI step using syft or assume it's provided
- GitHub Action error handling and edge cases
- Whether to include a marketplace icon/branding for the action

</decisions>
