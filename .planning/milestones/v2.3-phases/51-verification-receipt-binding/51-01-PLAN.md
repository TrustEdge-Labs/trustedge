Key facts:
- The non-postgres verify_handler generates `verification_id` as `format!("v_{}", uuid::Uuid::new_v4().simple())` -- fresh UUID each call
- Receipt JWS payload has `iat` (issued-at timestamp) set to `chrono::Utc::now().timestamp()`
- `manifest_digest` in receipt is computed via `compute_manifest_digest_blake3()` which does `serde_json::to_string(manifest)` then BLAKE3 hash
- Receipt claims are nested under `payload["receipt"]` in the JWS
- JWS format: base64url(header).base64url(payload).base64url(signature)
- Tests use `tower::ServiceExt::oneshot` -- each oneshot consumes the router, so create a fresh `create_test_app()` for each request

All tests use the existing `create_test_app()`, `build_signed_manifest()`, and `build_verify_body()` helpers. Use `base64::engine::general_purpose::URL_SAFE_NO_PAD as BASE64URL` (already imported in the module) to decode JWS payloads.

**Test 1: `sec_11_duplicate_submission_distinct_receipts`** (SEC-11)
- Generate one signing key, build one signed manifest + device_pub
- Build one verify body with `return_receipt: true`
- Create two separate `create_test_app()` instances (oneshot consumes the router)
- POST the identical body bytes to `/v1/verify` on each app
- Assert both return HTTP 200
- Parse both response JSON bodies
- Assert `resp1["verification_id"] != resp2["verification_id"]` -- each submission gets a unique ID
- Decode the JWS receipt from each response: split on '.', BASE64URL-decode parts[1] to get payload JSON
- Parse both payloads, extract `payload["receipt"]["verification_id"]`
- Assert the two receipt verification_ids differ
- Assert both receipts have `"iat"` fields (timestamps exist)

**Test 2: `sec_12_receipt_digest_bound_to_content`** (SEC-12)
- Generate two different signing keys (key_a, key_b)
- Build two different signed manifests: manifest_a with `"device_id": "device-alpha"` and manifest_b with `"device_id": "device-beta"` (different manifest content)
- Build verify bodies for each with `return_receipt: true`
- POST each to separate `create_test_app()` instances
- Assert both return HTTP 200 with receipts
- Decode both JWS receipt payloads
- Extract `manifest_digest` from each receipt
- Assert `digest_a != digest_b` -- different archive content produces different digest
- Also verify that each digest string starts with `"b3:"` (BLAKE3 prefix)

**Test 3: `sec_12_same_content_same_digest`** (SEC-12)
- Generate one signing key, build one signed manifest
- Build one verify body with `return_receipt: true`
- POST the same body to two separate `create_test_app()` instances
- Decode both JWS receipt payloads
- Extract `manifest_digest` from each
- Assert `digest_a == digest_b` -- same archive content produces same digest (deterministic binding)

Each test function is `#[tokio::test] async fn` returning `Result<()>`. Follow the existing code style: assert messages describe what property is being tested. Use `anyhow::Result` for the return type.
