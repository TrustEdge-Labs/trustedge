nginx's add_header directive in a location block REPLACES (not appends to) server-level headers. Every location block that uses add_header must repeat all security headers.

**deploy/nginx.conf** — Two location blocks need fixing:

1. `location ~*` (static assets, currently lines 19-22): Add these headers BEFORE the existing `add_header Cache-Control` line:
   ```
   add_header X-Content-Type-Options "nosniff" always;
   add_header X-Frame-Options "DENY" always;
   add_header Referrer-Policy "strict-origin-when-cross-origin" always;
   add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; connect-src 'self' http://localhost:3001; img-src 'self' data:; font-src 'self'" always;
   ```

2. `location /healthz` (currently lines 25-29): Add the same 4 headers BEFORE the existing `add_header Content-Type` line.

Also update the server-level CSP (line 11) to add `http://localhost:3001` to connect-src: change `connect-src 'self'` to `connect-src 'self' http://localhost:3001` (per D-02).

**deploy/nginx-ssl.conf.template** — Three location blocks need fixing, all in the same way:

1. SSL server `location ~*` (static assets, currently lines 26-29): Add the 4 security headers + HSTS before the Cache-Control header:
   ```
   add_header X-Content-Type-Options "nosniff" always;
   add_header X-Frame-Options "DENY" always;
   add_header Referrer-Policy "strict-origin-when-cross-origin" always;
   add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; connect-src 'self' http://localhost:3001; img-src 'self' data:; font-src 'self'" always;
   add_header Strict-Transport-Security "max-age=31536000" always;
   ```

2. SSL server `location /healthz` (currently lines 32-36): Same 4 security headers + HSTS before Content-Type header.

3. HTTP redirect server `location /healthz` (currently lines 45-49): Add only the 4 security headers (NO HSTS — this is the plain HTTP server) before Content-Type header.

Also update the SSL server-level CSP (line 17) to add `http://localhost:3001` to connect-src, same as nginx.conf.

Per D-01: Repeat headers inline in each block (no include snippet — these files are small and the repetition is the standard nginx pattern for this problem).
Per D-02: Adding `http://localhost:3001` directly to connect-src as the simplest approach that works for both dev and demo.

**deploy/docker-compose.yml** changes:

1. Update the top comment block: Replace "No .env file needed — all defaults are inline for demo use." with:
   ```
   # Copy deploy/.env.example to deploy/.env before first run.
   # See deploy/.env.example for all available settings.
   ```

2. `postgres` service: Replace inline `POSTGRES_PASSWORD: trustedge_dev` with env_file reference. Keep POSTGRES_USER and POSTGRES_DB inline (these are not secrets). The service should have:
   ```yaml
   postgres:
     image: postgres:16-alpine
     env_file: deploy/.env
     environment:
       POSTGRES_USER: trustedge
       POSTGRES_DB: trustedge
   ```
   Note: env_file values are overridden by `environment` values, and POSTGRES_PASSWORD from .env will be picked up automatically since it is not overridden in `environment`.

3. `platform-server` service: Replace inline `DATABASE_URL` with env_file reference. Keep PORT inline (not a secret):
   ```yaml
   platform-server:
     ...
     env_file: deploy/.env
     environment:
       PORT: "3001"
   ```
   DATABASE_URL will come from .env file.

**deploy/.env.example** changes:

Add a PostgreSQL credentials section with actual dev defaults (so `cp .env.example .env` works immediately for demo):

   ```
   # IMPORTANT: Change these for any non-local deployment.

   POSTGRES_PASSWORD=trustedge_dev
   DATABASE_URL=postgres://trustedge:trustedge_dev@postgres:5432/trustedge
   ```
   Remove the old commented-out `# DATABASE_URL=...` line that already exists (line 21) since we now have it as an active value. Keep the other existing sections (PORT, JWT_AUDIENCE, VITE_API_BASE, SSL) as they are.

**Update Usage comment** to:
