**Change 1 — Add `continue-on-error: true` to the job (per D-04, CI-04):**

On the `self-attestation:` job block, add `continue-on-error: true` immediately after the `needs: build-and-test` line:

```yaml
  self-attestation:
    runs-on: ubuntu-latest
    if: startsWith(github.ref, 'refs/tags/v')
    needs: build-and-test
    continue-on-error: true
    permissions:
      contents: write
```

**Change 2 — Replace unpinned syft install with anchore/sbom-action (per D-01, D-02, CI-02):**

Remove the two steps:
- `- name: Install syft` (the curl | sh step)
- `- name: Generate CycloneDX SBOM with syft` (the `syft ./target/release/trst` step)

Replace them with a single SHA-pinned action step, following the existing comment pattern `# vX.Y.Z`:

```yaml
      - name: Generate CycloneDX SBOM
        uses: anchore/sbom-action@e22c389904149dbc22b58101806040fa8d37a610 # v0.24.0
        with:
          path: ./target/release/trst
          format: cyclonedx-json
          output-file: trst-sbom.cdx.json
          upload-artifact: false
```

Note: `upload-artifact: false` prevents the action from uploading to the workflow run artifacts — we upload to the release ourselves in a later step.

**Change 3 — Rename ephemeral.pub to build.pub for clarity (per D-03, CI-03):**

In the `Generate ephemeral Ed25519 keypair` step, change `--out-pub ephemeral.pub` to `--out-pub build.pub`. The key file (`--out-key`) remains `ephemeral.key` — it is never uploaded:

```yaml
      - name: Generate ephemeral Ed25519 keypair
        run: |
          ./target/release/trst keygen \
            --out-key ephemeral.key \
            --out-pub build.pub \
            --unencrypted
```

In the `Create SBOM attestation` step, change `--device-pub ephemeral.pub` to `--device-pub build.pub`:

```yaml
      - name: Create SBOM attestation
        run: |
          ./target/release/trst attest-sbom \
            --binary ./target/release/trst \
            --sbom trst-sbom.cdx.json \
            --device-key ephemeral.key \
            --device-pub build.pub \
            --out trst.te-attestation.json \
            --unencrypted
```

**Change 4 — Upload build.pub as a release asset (per D-03, CI-01):**

In the `Upload attestation as release asset` step, upload both files:

```yaml
      - name: Upload attestation assets
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          gh release upload "${{ github.ref_name }}" \
            trst.te-attestation.json \
            build.pub \
            --clobber
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → release assets | The self-attestation job has `permissions: contents: write` and uses GITHUB_TOKEN to upload assets; a compromised action in the chain could tamper with assets |
| anchore/sbom-action (third-party) → runner filesystem | The sbom-action writes trst-sbom.cdx.json; a compromised action version could inject malicious SBOM content |
| ephemeral.key (private key) → runner memory | The ephemeral private key exists on disk during the CI run; must never be uploaded |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-79-01 | Tampering | anchore/sbom-action — third-party action injecting malicious SBOM | mitigate | SHA-pin at commit hash `e22c389904149dbc22b58101806040fa8d37a610` (not a mutable tag); pin is verified by GitHub's runner before execution. Matches existing pattern for all other actions in ci.yml. |
| T-79-02 | Information Disclosure | ephemeral.key uploaded as release asset | mitigate | Only `build.pub` (public key) is uploaded via `gh release upload`. `ephemeral.key` is never referenced in upload commands. Accepted: if runner is compromised at OS level, key material in memory is already lost — SHA-pinning reduces the attack surface to the runner itself. |
| T-79-03 | Tampering | Attacker replaces build.pub after upload to make a tampered binary appear valid | accept | Attacker would also need to re-sign trst.te-attestation.json with the replaced key. The SBOM binds a specific binary hash — a replaced binary with a replaced key still fails hash verification. Risk accepted: this is a CI attestation for portfolio visibility, not a regulated environment. |
| T-79-04 | Denial of Service | Attestation job failure blocks a legitimate release | mitigate | `continue-on-error: true` on the self-attestation job (per CI-04). A signing failure logs a warning but does not gate the release. |
| T-79-05 | Spoofing | GITHUB_TOKEN misuse — write-scoped token used beyond release upload | accept | `permissions: contents: write` is scoped to the self-attestation job only (job-level permissions override top-level `contents: read`). Token is never echoed or logged. Risk accepted: standard GitHub Actions supply chain trust model. |
</threat_model>
