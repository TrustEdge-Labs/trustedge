<objective>
Fix two independent platform defects introduced by the postgres feature:

1. (PLAT-01) `verify_handler` in postgres mode requires `Extension<OrgContext>` injected by auth middleware. Because `/v1/verify` is a public route (not behind auth middleware), Axum returns a 500 when the extension is missing. The fix is to make the extractor optional — `Option<Extension<OrgContext>>` — and operate tenant-agnostically when absent.

2. (PLAT-02) CORS allowed origins are hardcoded to `http://localhost:3000` and `http://localhost:8080`. Production deployments cannot override them. The fix is to read allowed origins from `CORS_ORIGINS` (comma-separated) with the existing localhost pair as fallback.

Purpose: Unblock postgres-mode verification for unauthenticated callers and enable production CORS configuration.
Output: Updated handlers.rs (optional OrgContext) and router.rs (env-driven CORS).
</objective>

REPLACE with:
```rust
// Read CORS allowed origins from CORS_ORIGINS env var (comma-separated).
// Falls back to localhost:3000,localhost:8080 when not set (dev default).
let cors_origins_raw = std::env::var("CORS_ORIGINS")
    .unwrap_or_else(|_| "http://localhost:3000,http://localhost:8080".to_string());

let allowed_origins: Vec<axum::http::HeaderValue> = cors_origins_raw
    .split(',')
    .map(|s| s.trim())
    .filter(|s| !s.is_empty())
    .filter_map(|s| {
        s.parse::<axum::http::HeaderValue>().map_err(|_| {
            tracing::warn!("CORS_ORIGINS: skipping invalid entry {:?}", s);
        }).ok()
    })
    .collect();

tracing::info!("CORS allowed origins: {:?}", allowed_origins);

let cors = CorsLayer::new()
    .allow_origin(allowed_origins)
    .allow_methods([axum::http::Method::GET, axum::http::Method::POST])
    .allow_headers([
        axum::http::header::CONTENT_TYPE,
        axum::http::header::AUTHORIZATION,
        axum::http::header::ACCEPT,
    ]);
```
